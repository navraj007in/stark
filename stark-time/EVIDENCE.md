## Known limitations

1. **The blocker** (see `BLOCKERS.md`): no owner-approved provider execution/linkage mechanism
   exists yet, so `Instant::now`, `UnixTimestamp::now`, and `Instant::elapsed` are unimplemented.
2. **No fake-provider injection seam** (see "Fake-provider results" above) — recorded per §18's
   own instruction, not redesigned.
3. **A discovered engine limitation, not a WP-TIME-A defect**: associated-function calls
   (`Type::function()`) only resolve within the same `.stark` file that declares the `impl`
   block; a call from a different file in the same package — even with a correct `use
   super::Type;` import or a fully-qualified `super::Type::function()` path — fails to resolve
   (`instance.method()` calls resolve correctly across files; only the `Type::function()` path
   does not). Reproduced minimally outside this package (a two-struct, two-file scratch package)
   before concluding it was not specific to `stark-time`'s code. Consequence: this package's
   tests live in `src/lib.stark` itself rather than a separate `src/tests.stark` (the pattern
   `stark-base64` uses), since `stark-base64` has no `impl` blocks and never hit this path. Also
   discovered in passing: `use super::{A, B};` (brace-list imports) fails to resolve either name,
   while separate `use super::A;` / `use super::B;` statements work — not relied upon anywhere in
   the final package, but recorded since it is the same general area. Not fixed here: WP-TIME-A
   §7.2 prohibits modifying name resolution.
4. Three of the four declared target triples have no build/test evidence from this session (see
   the cross-platform table).
